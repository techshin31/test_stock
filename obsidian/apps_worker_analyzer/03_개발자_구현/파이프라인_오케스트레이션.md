# 파이프라인 오케스트레이션

근거 코드: `apps/worker/analyzer/pipeline.py`

## 진입 함수

```text
apps.worker.__main__.run_analyze
  -> analyzer.pipeline.build_request
  -> analyzer.pipeline.run
```

`build_request()`는 문자열 날짜 옵션을 `date`로 변환하고 기본 날짜를 채운 뒤 `AnalysisRequest.validate()`를 호출한다.

## prepare_run

```text
prepare_run
  -> request.validate
  -> config.validate
  -> validate_source_readiness
  -> fetch_active_strategy
  -> fail_stale_analysis_runs
  -> _analysis_input_hash
  -> get_or_create_analysis_run
  -> AnalysisRunContext
```

준비 단계에서 run이 새로 생성되지 않으면 cache hit이다. 이때 `--publish`가 붙은 `analyze all`이면 기존 run을 발행할 수 있다.

## stage 실행 순서

| 순서 | progress label | 함수 |
|---:|---|---|
| 1 | 준비 | `prepare_run()` |
| 2 | 분기 FA | `refresh_quarterly_scores()` |
| 3 | 매크로 분석 | `run_macro_analysis()` |
| 4 | 섹터 분석 | `run_sector_analysis()` |
| 5 | 기업 선택 | `run_company_selection()` |
| 6 | 결과 검증 | `validate_run()` |
| 7 | 유니버스 발행 | `publish_universe()` |

`target`에 따라 뒤 단계는 실행하지 않는다.

## 예외 처리

각 stage는 `_run_stage()`로 감싼다. stage 내부 예외가 발생하면 해당 run은 즉시 `FAIL`로 바뀐다.

저장되는 실패 정보:

```text
status_code = FAIL
validation_summary = {"phase": phase, "target": context.target}
failure_reason = "{ExceptionType}:{message}" 최대 1000자
```

## 상태 업데이트

부분 target은 자체 요약을 `validation_summary`에 저장하고 PASS/WARNING을 결정한다.

| target | 상태 판정 |
|---|---|
| `macro` | input_quality, missing macro 결과 고려 |
| `sector` | 선택 업종 수와 input/macro 품질 고려 |
| `company` | 기업 선정 수 제한과 input/macro 품질 고려 |
| `all` | `validate_run()` 결과와 input/macro 품질 결합 |

## 개발 시 주의

- `validate_source_readiness()`는 run 생성 전 호출된다. readiness FAIL 분석 run은 남지 않는다.
- `get_or_create_analysis_run()`은 FAIL이 아닌 동일 input hash run을 재사용한다.
- stage 추가 시 input hash에 영향을 줄 설정은 `AnalyzerConfig.fingerprint`에 포함되어야 한다.
- Trader로 넘기는 공식 경로는 `publish_universe()`뿐이다.
