# 전략과 주문 흐름 해석

Trader는 종목을 새로 발굴하지 않는다. 이미 Analyzer가 발행한 `universe`를 읽고, 현재 보유 상태와 시장 가격/레짐을 반영해 오늘 주문할 수량을 계산한다.

```text
universe
  -> 보유 포지션 동기화
  -> 가격/레짐/신호 계산
  -> 목표 비중
  -> trade_plans
  -> orders/executions
```

## 투자 의사결정의 두 층

| 층 | 담당 | 의미 |
|---|---|---|
| 종목 후보 | Analyzer | FA 기반 ACTIVE/SELL_ONLY universe 발행 |
| 매매 실행 | Trader | 오늘의 목표 비중과 현재 보유 비중 차이를 주문 계획으로 변환 |

Trader는 `fa_company_results`를 직접 읽지 않는다. 투자 후보의 공식 운영 입력은 `universe`다.

## planner가 보는 종목

`planner`는 `fetch_universe_for_date()`로 기준일에 유효한 `ACTIVE`와 `SELL_ONLY` 종목을 읽는다.

| 상태 | Trader 해석 |
|---|---|
| `ACTIVE` | 신규 매수와 매도 모두 가능 |
| `SELL_ONLY` | 신규 매수는 막고 청산만 허용 |
| `REMOVED` | 운영 대상 아님 |

보유 중인데 universe에 없는 종목은 `SELL_ONLY`로 재등록된다. 이 장치는 이전에 보유한 종목이 새 universe에서 빠졌을 때 청산 계획을 만들기 위한 것이다.

## 가격과 신호 기준일

전략 신호 기준일은 `plan_date`가 아니라 직전 KRX 거래일이다.

```text
signal_date = previous_krx_trading_day(plan_date)
```

OHLCV는 yfinance로 2년치를 받아오고, `signal_date` 이후 데이터는 잘라낸다. 장전 계획이 당일 아직 완성되지 않은 가격 정보를 쓰지 않게 하기 위한 구조다.

## 목표 비중 계산

Planner가 계산하는 핵심 입력:

| 입력 | 출처 |
|---|---|
| KOSPI 레짐 | KOSPI 지수 OHLCV |
| 종목별 레짐 | 종목별 OHLCV |
| 종목별 신호 | `FaTaMomentumStrategy.make_signals_with_metadata` |
| 현재 비중 | `positions`와 최근 가격, 브로커 총자산 |
| 방어자산 | 전략 파라미터의 `bond_etf_code` |
| 기업 위험 | `company_risk_states` |

신호는 sparse한 트리거만 쓰지 않고 `position_after`를 사용한다. 이는 오늘 새로 universe에 들어온 종목도 현재 유지해야 할 목표 비중에 맞춰 주문 계획을 만들기 위한 선택이다.

## trade_plans의 투자 의미

`trade_plans`는 “오늘 무엇을 주문할지”뿐 아니라 “왜 주문하지 않았는지”도 남긴다.

| plan 상태 | 투자자 해석 |
|---|---|
| `PENDING` | 주문할 수량과 방향이 있음 |
| `SKIPPED` | 오늘 주문하지 않는다고 판단했고 사유를 기록 |
| `ORDERED` | 주문이 제출되었거나 일부 진행되어 추가 확인/재시도 필요 |
| `DONE` | 계획 수량 기준 완료 |

주문하지 않은 종목도 `SKIPPED`로 저장되므로 사후에 “universe에 있었는데 왜 매매가 없었는가”를 추적할 수 있다.

## 매매 사유 읽기

대표 `trade_reason_code`:

| 코드 | 의미 |
|---|---|
| `UPTREND_ENTRY1`, `UPTREND_ENTRY2` | 상승장 진입 신호에 따른 분할 매수 |
| `REBALANCE_BUY`, `REBALANCE_SELL` | 목표 비중과 현재 비중 차이 조정 |
| `DEFENSIVE_ALLOCATION` | 방어자산 비중 조정 |
| `DOWNTREND`, `TRANSITION_EXIT`, `DEADCROSS`, `ATR_STOP` | 위험 국면 또는 손절/축소 신호 |
| `NO_SIGNAL` | 오늘 목표 비중이 없음 |
| `BELOW_MIN_QTY` | 최소 주문 수량 미달 |
| `SELL_ONLY_BLOCKED` | SELL_ONLY 종목 추가 매수 차단 |
| `COMPANY_RISK_BLOCKED` | 기업 위험상태로 BUY 실행 차단 |

## 주문 가격 편차 제한

일부 매수/리밸런싱 사유는 ATR 기반 `price_deviation_limit`을 갖는다. executor는 현재 호가가 계획 가격 대비 허용 범위를 넘으면 해당 사이클의 주문을 건너뛰고 나중에 다시 볼 수 있다.

위험 축소 성격의 매도 사유는 제한이 `None`일 수 있다. 이 경우 가격 편차 때문에 지연시키지 않는 쪽에 가깝다.

## 체결과 손익 해석

`STATUS`의 당일 손익은 `executions.net_amount` 합이다. 매수는 음수, 매도는 양수로 저장되므로 이 값은 현금흐름 관점에 가깝다.

일일 손실 한도는 이 값이 아니라 전일 `balance_history.total_value`와 현재 브로커 총자산 차이를 기준으로 계산한다.
