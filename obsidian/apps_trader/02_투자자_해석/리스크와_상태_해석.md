# 리스크와 상태 해석

Trader의 리스크 관리는 한 곳에만 있지 않다. 실주문 gate, 일일 손실 한도, 기업 위험상태, SELL_ONLY, 호가 편차 제한, 브로커 주문상태 동기화가 함께 작동한다.

## 실주문 gate

실계좌 주문은 두 조건을 모두 요구한다.

```text
KIS_ENV=real
ALLOW_LIVE_ORDER=true
```

`KIS_ENV`가 `real`이 아니면 paper mode로 보고 gate를 통과시킨다. `real`인데 `ALLOW_LIVE_ORDER=true`가 아니면 Trader는 시작 단계에서 종료한다.

## 일일 손실 한도

executor 사이클은 매번 브로커 잔고를 조회하고 현재 총자산을 읽는다.

```text
current_total_value = broker.account.balance().output2[0].tot_evlu_amt
prev_total_value = latest balance_history.total_value
daily_net = current_total_value - prev_total_value
loss_limit = prev_total_value * DAILY_LOSS_LIMIT  # 기본 0.10 = 10%
```

기본값은 `DAILY_LOSS_LIMIT=0.10`이므로 전일 마감 총자산의 10%를 초과해 손실이 나면 사이클을 중단한다.

```text
daily_net < -loss_limit 이면 사이클 중단
```

`DAILY_LOSS_LIMIT`에 `500000`처럼 `1`보다 큰 값을 넣으면 비율이 아니라 50만원 고정 한도로 해석한다.

중요한 점:

- 기준은 당일 체결 현금흐름이 아니라 전일 마감 총자산 대비 현재 총자산이다.
- `0 < DAILY_LOSS_LIMIT <= 1`이면 전일 마감 총자산 대비 비율이다.
- `balance_history`가 없으면 손실 한도 체크를 건너뛴다.
- `balance_history`는 reconciler가 장마감에 저장한다.

## 기업 위험상태

`company_risk_states`의 최신 유효 상태가 `BLOCK_BUY` 또는 `SELL_ONLY`면 신규 매수를 막는다.

차단 위치는 두 곳이다.

| 위치 | 동작 |
|---|---|
| planner | 위험 종목을 sell-only set에 넣어 BUY 계획 생성을 막음 |
| executor | 실행 직전 다시 확인해 BUY plan을 `SKIPPED` + `COMPANY_RISK_BLOCKED` 처리 |

실행 직전 재확인이 있는 이유는 planner 이후 장중에 위험상태가 바뀔 수 있기 때문이다.

## universe 상태

| 상태 | 의미 | Trader 동작 |
|---|---|---|
| `ACTIVE` | 정상 매매 | 매수/매도 계획 가능 |
| `SELL_ONLY` | 청산 대기 | 추가 매수 차단, 매도만 허용 |
| `REMOVED` | 제거됨 | 운영 대상에서 제외 |

reconciler는 포지션 동기화 이후 SELL_ONLY 종목의 보유 수량이 0이면 `REMOVED`로 전환한다.

## plan 상태

| 상태 | 의미 | executor/reconciler 관점 |
|---|---|---|
| `PENDING` | 실행 전 또는 재시도 가능 | 실행 대상 |
| `ORDERED` | 주문 제출/부분 진행/미확정 | 실행 대상 또는 reconcile 필요 |
| `DONE` | 계획 완료 | 닫힌 상태 |
| `SKIPPED` | 오늘 주문 없음 | 닫힌 상태 |
| `CANCELLED` | 인터럽트 등으로 취소 | 닫힌 상태로 취급 가능하지만 원인 확인 필요 |

`PENDING/ORDERED`가 0개면 executor는 루프를 종료하고, reconciler는 15:40 대기를 건너뛸 수 있다.

## order 상태

`orders`는 브로커 주문 한 건의 상태를 저장한다.

| 상태 | 의미 |
|---|---|
| `SUBMITTED` | 로컬 주문 행 생성, 증권사 제출 단계 |
| `ACCEPTED` | 증권사 주문번호 연결 |
| `PARTIAL` | 일부 체결 |
| `FILLED` | 전량 체결 |
| `CANCELLED` | 취소 |
| `REJECTED` | 제출 전/제출 중 오류 등 거부 |

`order_status_history`는 상태 변화와 브로커 조회 이벤트를 append-only로 남긴다.

## 호가와 수량 제한

executor는 주문 전 다음을 확인한다.

| 검사 | BUY | SELL |
|---|---|---|
| 가격 편차 | 현재 매도호가가 계획가보다 너무 높으면 skip | 현재 매수호가가 계획가보다 너무 낮으면 skip |
| 가능 수량 | KIS 매수가능수량으로 제한 | KIS 매도가능수량으로 제한 |
| 오더북 가시 수량 | 최우선 호가 잔량의 일부만 child order로 사용 | 동일 |
| child 수량 | `max_child_qty` 이하 | 동일 |

가격이나 가능 수량 조건으로 건너뛴 plan은 즉시 닫히지 않을 수 있다. 다음 사이클에서 다시 확인하는 구조다.

## 투자자 관점 확인 순서

1. `universe`의 ACTIVE/SELL_ONLY 상태가 의도와 맞는지 본다.
2. `trade_plans`에서 `PENDING`, `SKIPPED`, `ORDERED`, `DONE` 분포를 본다.
3. `trade_reason_code`로 주문 또는 미주문 사유를 본다.
4. `orders`와 `order_status_history`에서 브로커 주문 진행 상태를 본다.
5. `executions`에서 실제 체결 수량과 가격을 본다.
6. `positions`와 `balance_history`로 장마감 상태를 확인한다.
