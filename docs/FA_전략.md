# FA 전략 : 분기별 FA 분석

## 1. 이 문서의 목적

이 문서는 FA(Fundamental Analysis)를 백테스트에 바로 넣는 매매 규칙이 아니라, **분기마다 기업의 기초체력이 어떻게 변했는지 자동으로 판정하는 분석 체계**로 다시 정의한다.

세부 문서는 아래처럼 분리한다.


| 문서                          | 역할                       |
| --------------------------- | ------------------------ |
| `docs/FA_학습가이드.md`          | 처음 공부할 때 보는 핵심 개념과 학습 순서 |
| `docs/FA_전략.md`             | 분기별 FA 분석의 방향과 이유        |
| `docs/FA_점수체계_자동화_검증.ipynb` | 점수 계산과 검증을 실행하는 최소 노트북   |


현재 프로젝트의 백테스트는 이미 별도 단계로 진행되고 있다. 따라서 지금 필요한 것은 백테스트 엔진을 다시 만드는 것이 아니라, 백테스트가 사용할 수 있는 신뢰 가능한 FA 입력값을 만드는 것이다.

정리하면 역할은 다음과 같이 분리한다.


| 단계            | 역할                         |
| ------------- | -------------------------- |
| 분기별 FA 분석     | 기업별 재무 상태, 변화, 위험을 점수화한다   |
| 리밸런싱          | 어느 시점에 어떤 FA 점수를 사용할지 결정한다 |
| vectorBT 백테스트 | FA 결과를 이용해 포트폴리오 성과를 검증한다  |


즉 이번 문서의 중심은 `target_weight_matrix`가 아니라 다음 테이블이다.

```text
company_quarter_fa
```

이 테이블은 다음 질문에 답해야 한다.

```text
특정 기업은 특정 분기에 재무적으로 좋아졌는가, 나빠졌는가, 위험해졌는가?
```

## 2. 왜 분기별 FA로 다시 정의해야 하는가

기존 연간 FA 분석은 기업의 장기 체력을 보기에는 좋지만, 백테스트와 자동화에는 한계가 있다.

첫째, 연간 데이터는 반응이 너무 느리다. 기업의 실적 악화나 회복은 분기 단위로 먼저 나타나는 경우가 많다. 연간 기준만 사용하면 이미 주가와 시장 평가가 크게 움직인 뒤에야 점수가 바뀔 수 있다.

둘째, 백테스트에서 시점 오류가 생기기 쉽다. 예를 들어 2025년 사업보고서 데이터는 2025년 말에 존재하는 정보가 아니다. 실제로는 2026년에 공시된 뒤에야 사용할 수 있다. 따라서 단순히 회계연도 기준으로 점수를 붙이면 룩어헤드 바이어스가 발생할 수 있다.

셋째, 자동화하려면 분석 단위가 명확해야 한다. 사람이 리포트를 읽고 판단하는 방식은 유연하지만, 시스템은 다음 기준이 있어야 반복 가능하다.


| 필요한 기준   | 설명                         |
| -------- | -------------------------- |
| 분석 대상 기간 | 이 점수가 어느 회계 분기를 평가한 것인지    |
| 사용 가능일   | 이 정보를 실제 투자 시점에 알 수 있었는지   |
| 계산 지표    | 어떤 재무지표로 점수를 만들었는지         |
| 제외 사유    | 왜 특정 종목을 분석 또는 투자 대상에서 뺐는지 |
| 점수 구조    | 좋은 기업과 위험한 기업을 어떻게 구분했는지   |


따라서 FA 자동화의 출발점은 “좋은 종목을 고르는 공식”이 아니라, **분기별로 재현 가능한 기업 상태 스냅샷을 만드는 것**이어야 한다.

## 3. 핵심 개념 분리

분기 FA를 만들 때 가장 먼저 분리해야 하는 날짜는 세 가지다.


| 개념               | 의미                          | 예시            |
| ---------------- | --------------------------- | ------------- |
| `fiscal_quarter` | 재무제표가 설명하는 회계 분기            | 2025Q1        |
| `available_date` | 해당 재무제표를 투자자가 사용할 수 있게 된 날짜 | 2025-05-15 이후 |
| `rebalance_date` | 포트폴리오를 다시 구성하는 날짜           | 2025-06-30    |


이 셋을 섞으면 안 된다.

분기 FA 분석은 `fiscal_quarter` 기준으로 기업 상태를 계산한다. 하지만 백테스트에서는 반드시 `available_date <= rebalance_date` 조건을 만족하는 자료만 사용해야 한다.

예시는 다음과 같다.

```text
2025Q1 실적
공시 또는 사용 가능일: 2025-05-15
리밸런싱 날짜: 2025-03-31
사용 가능 여부: 불가능

2025Q1 실적
공시 또는 사용 가능일: 2025-05-15
리밸런싱 날짜: 2025-06-30
사용 가능 여부: 가능
```

이 원칙이 분기 FA 자동화의 가장 중요한 안전장치다.

## 4. 분기별 FA 테이블 정의

`company_quarter_fa`는 기업, 회계 분기, 사용 가능일을 기준으로 만들어지는 분석 테이블이다.

최소 컬럼은 다음과 같다.


| 컬럼                   | 설명                 |
| -------------------- | ------------------ |
| `stock_code`         | 종목 코드              |
| `company_name`       | 기업명                |
| `sector_code`        | WICS 섹터 코드         |
| `sector_name`        | WICS 섹터명           |
| `fiscal_year`        | 회계연도               |
| `fiscal_quarter`     | 회계 분기              |
| `report_code`        | DART 보고서 코드        |
| `period_end`         | 해당 재무제표 기준일        |
| `available_date`     | 백테스트에서 사용 가능해지는 날짜 |
| `revenue`            | 매출액                |
| `operating_income`   | 영업이익               |
| `net_income`         | 당기순이익              |
| `total_assets`       | 자산총계               |
| `total_equity`       | 자본총계               |
| `total_liabilities`  | 부채총계               |
| `operating_cashflow` | 영업활동현금흐름           |
| `market_cap`         | 기준 시점 시가총액         |
| `fa_score`           | 최종 FA 점수           |
| `fa_grade`           | 점수 구간 등급           |
| `excluded_reason`    | 제외 사유              |


이 테이블은 최종 투자 결과물이 아니라, 백테스트와 리밸런싱이 참조할 수 있는 **FA 원장**이다.

## 5. 점수 구조

분기 FA 점수는 단순한 현재 상태 점수만으로 만들면 부족하다. 분기 분석의 장점은 “상태”와 “변화”를 함께 볼 수 있다는 점이다.

따라서 v1 점수는 다음 구조로 시작한다.

```text
quarter_fa_score
= level_score
+ change_score
- risk_penalty
```

총점은 100점 기준으로 정규화한다.


| 점수 축           | 배점  | 의미                         |
| -------------- | --- | -------------------------- |
| `level_score`  | 60  | 현재 기업의 재무 체력이 좋은가          |
| `change_score` | 30  | 전년 동기 또는 직전 분기 대비 좋아지고 있는가 |
| `risk_penalty` | 10  | 투자 대상에서 감점해야 할 위험이 있는가     |


이 구조를 쓰는 이유는 명확하다.

`level_score`만 보면 이미 좋은 기업만 계속 높게 나온다. 반대로 `change_score`만 보면 일시적 턴어라운드나 기저효과에 과하게 반응할 수 있다. 그래서 현재 체력 60, 변화 30, 위험 10의 구조로 시작한다.

### 5.1 점수체계 상세 정의

v1 점수체계는 아래처럼 나눈다.

```text
fa_score = level_score + change_score - risk_penalty
```

각 점수는 역할이 다르다.


| 점수             | 역할       | 기본 배점 | 해석                        |
| -------------- | -------- | ----- | ------------------------- |
| `level_score`  | 현재 상태 평가 | 60    | 지금 재무 체력이 좋은 기업인가         |
| `change_score` | 변화 방향 평가 | 30    | 전년 동기 또는 직전 기간보다 좋아지고 있는가 |
| `risk_penalty` | 하방 위험 차감 | 10    | 점수가 좋아 보여도 피해야 할 위험이 있는가  |


즉 “좋은 기업”만 찾지 않는다. **좋은 상태**, **좋아지는 방향**, **피해야 할 위험**을 분리해서 계산한다.

#### Level Score 상세

`level_score`는 현재 분기의 재무 체력을 평가한다.

v1에서는 섹터별 사업 구조 차이를 반영하기 위해 같은 `fiscal_quarter`와 같은 WICS 섹터 안에서 상대평가를 기본으로 한다.


| 하위 항목 | 예시 원천 변수 | 원천 변수 의미 | 계산 예시 | 방향 | 기본 비중 |
|---|---|---|---|---|---:|
| 수익성 | `revenue`, `operating_income`, `net_income`, `total_equity` | 매출액, 영업이익, 당기순이익, 자본총계 | 영업이익률, ROE | 높을수록 좋음 | 20 |
| 안정성 | `total_liabilities`, `total_equity`, `current_assets`, `current_liabilities` | 부채총계, 자본총계, 유동자산, 유동부채 | 부채비율, 유동비율 | 부채비율은 낮을수록, 유동비율은 높을수록 좋음 | 15 |
| 현금흐름 | `operating_cashflow`, `revenue`, `net_income` | 영업활동현금흐름, 매출액, 당기순이익 | OCF/매출, OCF/순이익 | 높을수록 좋음 | 15 |
| 밸류에이션 | `market_cap`, `net_income`, `total_equity` | 시가총액, 당기순이익, 자본총계 | PER, PBR, 이익 대비 시총 | 낮을수록 좋음 | 10 |


합계는 60점이다.

처음 구현에서는 모든 지표가 준비되지 않을 수 있다. 이 경우 사용 가능한 지표만으로 재정규화하되, 반드시 `score_confidence`를 함께 계산한다.

```text
level_score_confidence
= 실제 사용된 level 하위 지표 수 / 필요한 level 하위 지표 수
```

#### Change Score 상세

`change_score`는 기업이 좋아지고 있는지를 평가한다.

분기 분석에서는 현재 값보다 변화 방향이 중요할 때가 많다. 다만 직전 분기 대비 변화는 계절성에 흔들릴 수 있으므로 v1에서는 YoY를 우선한다.


| 하위 항목 | 예시 원천 변수 | 원천 변수 의미 | 계산 예시 | 방향 | 기본 비중 |
|---|---|---|---|---|---:|
| 매출 성장 | `revenue`, `revenue_yoy_base` | 이번 분기 매출액, 전년 동기 매출액 | 이번 분기 매출 / 전년 동기 매출 - 1 | 높을수록 좋음 | 10 |
| 영업이익 성장 | `operating_income`, `operating_income_yoy_base` | 이번 분기 영업이익, 전년 동기 영업이익 | 이번 분기 영업이익 / 전년 동기 영업이익 - 1 | 높을수록 좋음 | 8 |
| 수익성 개선 | `operating_margin`, `operating_margin_yoy_base` | 이번 분기 영업이익률, 전년 동기 영업이익률 | 이번 분기 OPM - 전년 동기 OPM | 높을수록 좋음 | 6 |
| 현금흐름 개선 | `operating_cashflow`, `operating_cashflow_yoy_base` | 이번 분기 영업활동현금흐름, 전년 동기 영업활동현금흐름 | 이번 분기 OCF - 전년 동기 OCF | 높을수록 좋음 | 6 |


합계는 30점이다.

변화 점수는 아래 순서로 계산한다.

```text
1순위: YoY, 전년 동기 대비
2순위: QoQ, 직전 분기 대비
3순위: 최근 4개 분기 추세
```

YoY 기준값이 없으면 QoQ나 최근 4개 분기 추세를 보조로 쓸 수 있다. 다만 대체 기준을 쓴 경우에는 `change_score_confidence`를 낮춘다.

```text
change_score_confidence
= 실제 사용된 change 하위 지표 수 / 필요한 change 하위 지표 수
```

#### Risk Penalty 상세

`risk_penalty`는 좋은 점수에 브레이크를 거는 장치다.

자동화에서는 상위 점수 종목을 계속 새로 밀어 넣는 것보다, 위험한 종목이 상위권에 들어오는 것을 막는 것이 더 중요하다. 따라서 위험 조건은 별도 차감 점수로 둔다.


| 위험 항목 | 예시 원천 변수 | 원천 변수 의미 | 조건 예시 | 처리 | 최대 감점 |
|---|---|---|---|---|---:|
| 자본잠식 또는 자본 취약 | `total_equity` | 자본총계 | 자본총계 0 이하 | 원칙적으로 제외 | 10 |
| 현금흐름 악화 | `operating_cashflow`, `net_income` | 영업활동현금흐름, 당기순이익 | 이익은 양수인데 OCF가 음수 | 감점 | 3 |
| 부채 부담 급증 | `debt_ratio`, `debt_ratio_prev` | 이번 분기 부채비율, 이전 비교 기준 부채비율 | 부채비율 급등 | 감점 | 2 |
| 성장의 질 악화 | `revenue`, `operating_income`, `operating_cashflow` | 매출액, 영업이익, 영업활동현금흐름 | 매출 증가 + 이익/OCF 동시 악화 | 감점 | 3 |
| 주요 지표 결측 | 핵심 재무 컬럼 | 점수 계산에 필요한 필수 재무 항목 | 필수 지표 부족 | 감점 또는 제외 | 2 |


위험 점수는 최대 10점까지 차감한다.

```text
fa_score = level_score + change_score - min(risk_penalty, 10)
```

단, 자본총계 0 이하처럼 투자 대상에서 제외해야 하는 조건은 단순 감점이 아니라 `excluded_reason`을 남기고 제외한다.

#### 섹터별 보정

모든 섹터에 같은 하위 지표를 같은 비중으로 적용하면 왜곡이 생긴다.

예를 들어 금융은 부채비율이 일반 제조업과 다르게 해석되고, 건강관리의 적자 바이오는 전통적인 수익성 지표만으로 평가하기 어렵다.

따라서 v1은 공통 구조를 유지하되, 섹터별로 하위 지표 비중을 조정할 수 있게 둔다.


| 섹터/모델       | 우선하는 점수축           | 이유                     |
| ----------- | ------------------ | ---------------------- |
| 일반 산업       | 수익성, 안정성, 현금흐름, 성장 | 기본 FA 평가축이 비교적 잘 작동    |
| 금융          | 수익성, 자본효율, 주주환원    | 일반 부채비율 해석이 왜곡될 수 있음   |
| 유틸리티        | 현금흐름, 안정성          | 성장보다 안정적 운영과 부채 부담이 중요 |
| 에너지/소재      | 현금흐름, 안정성, 사이클 변화  | 업황 변동성이 큼              |
| 건강관리 흑자     | 성장성, 수익성, 현금흐름     | 제품 매출과 이익 성장 확인 가능     |
| 건강관리 적자/바이오 | 재무생존성, 이벤트, 비용통제   | 전통적 이익 지표만으로 평가하기 어려움  |


섹터별 보정을 하더라도 최종 출력은 동일한 구조를 유지한다.

```text
level_score
change_score
risk_penalty
fa_score
score_confidence
excluded_reason
```

#### Score Confidence

`score_confidence`는 점수 자체만큼 중요하다.

같은 80점이라도 모든 지표가 갖춰진 80점과, 일부 지표만으로 재정규화된 80점은 신뢰도가 다르다.

v1에서는 아래처럼 계산한다.

```text
score_confidence
= 사용된 하위 지표 수 / 필요한 하위 지표 수
```

가능하면 축별 신뢰도도 함께 남긴다.


| 컬럼                        | 의미                |
| ------------------------- | ----------------- |
| `level_score_confidence`  | 현재 상태 점수의 데이터 충족도 |
| `change_score_confidence` | 변화 점수의 데이터 충족도    |
| `risk_score_confidence`   | 위험 감점 판단의 데이터 충족도 |
| `score_confidence`        | 전체 점수의 종합 신뢰도     |


하방 방어 자동화에서는 `fa_score`가 높더라도 `score_confidence`가 낮으면 신규 결과로 적극 사용하지 않는다.

#### 점수 계산 절차

점수는 아래 순서로 계산한다.

```text
1. 원천 재무값 정리
2. 파생 지표 계산
3. 섹터 내 percentile 계산
4. 하위 항목 점수 계산
5. level_score, change_score, risk_penalty 계산
6. score_confidence 계산
7. fa_score 계산
8. fa_grade와 fallback 판정에 사용할 검증값 저장
```

##### 1. 원천 재무값 정리

먼저 `company_quarter_fa`의 원천 컬럼을 숫자형으로 정리한다.

```text
revenue
operating_income
net_income
total_assets
total_equity
total_liabilities
current_assets
current_liabilities
operating_cashflow
market_cap
```

0으로 나눌 수 있는 계산은 결과를 결측으로 둔다. 임의로 0점 처리하지 않는다. 0점은 “나쁜 값”이고 결측은 “판단 불가”이므로 둘을 구분해야 한다.

##### 2. 파생 지표 계산

원천값에서 아래 파생 지표를 만든다.

| 지표 | 한글 의미 | 계산식 | 방향 |
|---|---|---|---|
| `operating_margin` | 영업이익률 | `operating_income / revenue` | 높을수록 좋음 |
| `roe` | 자기자본이익률 | `net_income / total_equity` | 높을수록 좋음 |
| `debt_ratio` | 부채비율 | `total_liabilities / total_equity` | 낮을수록 좋음 |
| `current_ratio` | 유동비율 | `current_assets / current_liabilities` | 높을수록 좋음 |
| `ocf_to_revenue` | 매출 대비 영업현금흐름 비율 | `operating_cashflow / revenue` | 높을수록 좋음 |
| `ocf_to_net_income` | 순이익 대비 영업현금흐름 비율 | `operating_cashflow / net_income` | 높을수록 좋음 |
| `per_proxy` | 시가총액 대비 순이익 배수 | `market_cap / net_income` | 낮을수록 좋음 |
| `pbr_proxy` | 시가총액 대비 자본 배수 | `market_cap / total_equity` | 낮을수록 좋음 |
| `revenue_growth_yoy` | 전년 동기 대비 매출 성장률 | `revenue / revenue_yoy_base - 1` | 높을수록 좋음 |
| `operating_income_growth_yoy` | 전년 동기 대비 영업이익 성장률 | `operating_income / operating_income_yoy_base - 1` | 높을수록 좋음 |
| `operating_margin_change_yoy` | 전년 동기 대비 영업이익률 개선폭 | `operating_margin - operating_margin_yoy_base` | 높을수록 좋음 |
| `operating_cashflow_change_yoy` | 전년 동기 대비 영업현금흐름 개선폭 | `operating_cashflow - operating_cashflow_yoy_base` | 높을수록 좋음 |

PER, PBR은 실제 주가와 주식 수 기반 지표가 준비되기 전까지 `market_cap` 기반 프록시로 둔다. 프록시를 쓴 경우에는 해당 지표의 출처를 남긴다.

##### 3. 섹터 내 percentile 계산

각 지표는 같은 `fiscal_quarter`와 같은 WICS 섹터 안에서 percentile로 바꾼다.

```text
group_key = fiscal_quarter + wics_large
```

높을수록 좋은 지표는 그대로 percentile을 쓴다.

```text
metric_score = percentile_rank(metric)
```

낮을수록 좋은 지표는 역방향 percentile을 쓴다.

```text
metric_score = percentile_rank(-metric)
```

점수 범위는 `0~1`로 둔다.

```text
0.0 = 섹터 내 최하위권
1.0 = 섹터 내 최상위권
```

동점값은 같은 percentile을 부여한다. 특히 이벤트 수, 결측 대체값, 0값처럼 같은 값이 많이 나오는 지표는 순서대로 임의 등수를 나누면 안 된다.

##### 4. 하위 항목 점수 계산

하위 항목은 관련 metric score의 평균으로 계산한다.

```text
profitability_subscore
= mean(operating_margin_score, roe_score)

stability_subscore
= mean(debt_ratio_score, current_ratio_score)

cashflow_subscore
= mean(ocf_to_revenue_score, ocf_to_net_income_score)

valuation_subscore
= mean(per_proxy_score, pbr_proxy_score)

revenue_growth_subscore
= revenue_growth_yoy_score

operating_income_growth_subscore
= operating_income_growth_yoy_score

margin_improvement_subscore
= operating_margin_change_yoy_score

cashflow_improvement_subscore
= operating_cashflow_change_yoy_score
```

사용 가능한 지표가 없으면 해당 하위 항목은 결측으로 둔다.

##### 5. Level, Change, Risk 계산

`level_score`는 하위 항목 점수에 배점을 곱해 계산한다.

```text
level_score
= profitability_subscore * 20
+ stability_subscore * 15
+ cashflow_subscore * 15
+ valuation_subscore * 10
```

`change_score`는 변화 하위 항목에 배점을 곱해 계산한다.

```text
change_score
= revenue_growth_subscore * 10
+ operating_income_growth_subscore * 8
+ margin_improvement_subscore * 6
+ cashflow_improvement_subscore * 6
```

결측 항목이 있으면 사용 가능한 항목의 배점만 기준으로 재정규화한다.

예를 들어 `valuation_subscore`가 없고 나머지 50점만 계산 가능하면:

```text
level_score = raw_level_score / available_level_weight * 60
```

단, 이때 `score_confidence`가 낮아진다. 결측 재정규화는 점수를 계산하기 위한 편의일 뿐, 신뢰도를 보장하지 않는다.

`risk_penalty`는 조건별 감점을 합산한다.

```text
risk_penalty
= equity_risk_penalty
+ cashflow_risk_penalty
+ debt_risk_penalty
+ growth_quality_penalty
+ missing_metric_penalty
```

최대 10점으로 제한한다.

```text
risk_penalty = min(risk_penalty, 10)
```

제외 조건은 감점보다 우선한다.

```text
if total_equity <= 0:
    excluded_reason = "non_positive_equity"
    fa_score = null
```

##### 6. Score Confidence 계산

신뢰도는 실제 사용된 배점 비율로 계산한다.

```text
level_score_confidence
= available_level_weight / total_level_weight

change_score_confidence
= available_change_weight / total_change_weight

risk_score_confidence
= available_risk_checks / total_risk_checks
```

전체 신뢰도는 축별 가중 평균으로 둔다.

```text
score_confidence
= level_score_confidence * 0.6
+ change_score_confidence * 0.3
+ risk_score_confidence * 0.1
```

##### 7. 최종 점수 계산

최종 점수는 0~100점으로 둔다.

```text
fa_score = level_score + change_score - risk_penalty
fa_score = max(0, min(fa_score, 100))
```

등급은 아래처럼 나눈다.

| 점수 구간 | `fa_grade` |
|---|---|
| 80 이상 | `A` |
| 60 이상 80 미만 | `B` |
| 40 이상 60 미만 | `C` |
| 20 이상 40 미만 | `D` |
| 20 미만 | `E` |

하방 방어 운영에서는 `fa_grade`만 보지 않는다. 반드시 `score_confidence`와 `excluded_reason`을 함께 본다.

##### 8. 계산 결과에 반드시 남길 컬럼

최소한 아래 컬럼은 결과에 남긴다.

| 컬럼 | 의미 |
|---|---|
| `level_score` | 현재 상태 점수 |
| `change_score` | 변화 방향 점수 |
| `risk_penalty` | 위험 감점 |
| `fa_score` | 최종 점수 |
| `fa_grade` | 점수 구간 |
| `level_score_confidence` | 현재 상태 점수 신뢰도 |
| `change_score_confidence` | 변화 점수 신뢰도 |
| `risk_score_confidence` | 위험 판단 신뢰도 |
| `score_confidence` | 전체 점수 신뢰도 |
| `excluded_reason` | 제외 사유 |
| `score_model` | 사용된 섹터별 점수 모델 |
| `source_metric_coverage` | 사용 가능한 원천 지표 비율 |

## 6. Level Score

`level_score`는 현재 분기 기준 기업의 절대적 또는 섹터 상대적 체력을 평가한다.


| 항목    | 예시 지표                | 방향        |
| ----- | -------------------- | --------- |
| 수익성   | 영업이익률, ROE           | 높을수록 좋음   |
| 안정성   | 부채비율, 자본총계, 유동비율     | 안정적일수록 좋음 |
| 현금흐름  | 영업현금흐름, OCF/매출       | 높을수록 좋음   |
| 밸류에이션 | PER, PBR, 시가총액 대비 이익 | 낮을수록 좋음   |


v1에서는 절대 기준보다 섹터 내 상대 순위를 우선한다.

```text
같은 fiscal_quarter + 같은 WICS 섹터 안에서 percentile rank 계산
```

섹터별 사업 구조가 다르기 때문이다. 금융, 제조, 바이오, 플랫폼 기업을 하나의 절대 기준으로 비교하면 점수가 왜곡될 수 있다.

---

## 7. Change Score

`change_score`는 분기 FA에서 가장 중요한 부분이다. 자동화된 FA가 연간 분석보다 의미 있으려면 “좋은 기업인가”뿐 아니라 “좋아지고 있는 기업인가”를 잡아야 한다.

우선순위는 전년 동기 대비 변화다.


| 항목       | 계산 방향                   |
| -------- | ----------------------- |
| 매출 성장률   | 이번 분기 매출 / 전년 동기 매출     |
| 영업이익 성장률 | 이번 분기 영업이익 / 전년 동기 영업이익 |
| 영업이익률 개선 | 이번 분기 OPM - 전년 동기 OPM   |
| 현금흐름 개선  | 이번 분기 OCF - 전년 동기 OCF   |


직전 분기 대비 변화도 참고할 수 있지만, v1의 중심 기준으로 두지는 않는다. 계절성이 있는 업종에서는 직전 분기 대비 비교가 오히려 왜곡될 수 있기 때문이다.

따라서 기본 비교 순서는 다음과 같다.

```text
1순위: YoY, 전년 동기 대비
2순위: QoQ, 직전 분기 대비
3순위: 최근 4개 분기 추세
```

---

## 8. Risk Penalty

`risk_penalty`는 좋은 점수를 깎기 위한 장치다. 점수 상위 종목 중에서도 재무적으로 피해야 할 종목이 있을 수 있기 때문이다.

v1 위험 조건은 다음과 같이 둔다.


| 위험 조건                      | 처리           |
| -------------------------- | ------------ |
| 자본총계 0 이하                  | 원칙적으로 제외     |
| 영업현금흐름 급격한 악화              | 감점           |
| 부채비율 급등                    | 감점           |
| 매출은 증가했지만 이익과 현금흐름이 동시에 악화 | 감점           |
| 적자 지속 기업                   | 섹터별 별도 기준 필요 |
| 주요 재무 항목 결측                | 감점 또는 제외     |


특히 건강관리, 바이오, 성장주 성격의 기업은 단순 적자 여부만으로 제외하면 안 된다. 이런 섹터는 기존 `docs/FA.md`의 업종별 기준처럼 재무생존성, 현금 보유, 현금 소진 속도 기준을 따로 둬야 한다.

---

## 9. 제외 규칙

점수를 계산하기 전에 분석 불가능하거나 투자 부적합한 종목을 분리한다.


| 조건                  | 처리           |
| ------------------- | ------------ |
| 종목 코드 또는 DART 매핑 없음 | 제외           |
| 핵심 재무제표 항목 결측       | 제외 또는 낮은 신뢰도 |
| 자본총계 0 이하           | 제외           |
| 시가총액 계산 불가          | 제외           |
| 가격 데이터 없음           | 백테스트 대상 제외   |
| WICS 섹터 없음          | 섹터 상대평가 제외   |


중요한 점은 제외 종목을 조용히 버리면 안 된다는 것이다.

반드시 `excluded_reason`을 남겨야 한다. 그래야 나중에 성과가 나쁠 때 점수체계 문제인지, 데이터 결측 문제인지, 필터가 과도했는지 확인할 수 있다.

## 10. 리밸런싱과의 연결

분기 FA 분석은 리밸런싱 그 자체가 아니다.

분기 FA 분석은 다음 산출물을 만든다.

```text
company_quarter_fa
```

리밸런싱 단계는 특정 날짜에 이 테이블에서 사용 가능한 최신 FA만 고른다.

```text
rebalance_date 기준
available_date <= rebalance_date 인 자료 중
가장 최신 fiscal_quarter 선택
```

그 결과가 백테스트용 스냅샷이 된다.

```text
fa_snapshot_by_rebalance_date
```

vectorBT는 이 스냅샷을 직접 해석하지 않는다. vectorBT에 들어가기 전, 별도 전략 계층에서 종목 선택과 비중 배분을 수행한 뒤 `target_weight_matrix`로 변환한다.

흐름은 다음과 같다.

```text
company_quarter_fa
→ fa_snapshot_by_rebalance_date
→ selected_stocks
→ target_weight_matrix
→ vectorBT backtest
```

## 11. 자동화 전략

자동화는 한 번에 매매까지 연결하지 않는다. 순서를 나누어 검증한다.

### 1단계: 분기 FA 원장 생성

목표는 기업별 분기 재무지표를 안정적으로 계산하는 것이다.

산출물:

```text
company_quarter_fa
```

검증 기준:


| 검증 항목  | 설명                                   |
| ------ | ------------------------------------ |
| 분기 수   | 2021Q1부터 최신 분기까지 누락 없이 생성되는가         |
| 종목 수   | WICS, DART 매핑 종목 수가 기대 범위에 들어오는가     |
| 결측률    | 주요 재무지표 결측률이 과도하지 않은가                |
| 날짜 정합성 | `available_date`가 `period_end`보다 뒤인가 |


### 2단계: 점수 계산 검증

목표는 점수가 상식적으로 움직이는지 확인하는 것이다.

검증 기준:


| 검증 항목     | 설명                            |
| --------- | ----------------------------- |
| 섹터별 상위 종목 | 사람이 봐도 납득 가능한 기업이 나오는가        |
| 점수 변동     | 실적 개선 기업의 점수가 실제로 올라가는가       |
| 위험 감점     | 자본잠식, 현금흐름 악화 기업이 감점되는가       |
| 결측 처리     | 데이터 없는 기업이 부당하게 높은 점수를 받지 않는가 |


### 3단계: 리밸런싱 스냅샷 생성

목표는 백테스트 시점에서 실제로 알 수 있었던 FA만 선택하는 것이다.

검증 기준:


| 검증 항목       | 설명                                              |
| ----------- | ----------------------------------------------- |
| 룩어헤드 방지     | `available_date <= rebalance_date` 조건이 항상 지켜지는가 |
| 최신 분기 선택    | 사용 가능한 분기 중 가장 최신 자료를 쓰는가                       |
| 종목 universe | 해당 날짜에 투자 가능한 종목만 남는가                           |


### 4단계: vectorBT 연결

목표는 FA 점수를 백테스트 입력으로 변환하는 것이다.

산출물:

```text
target_weight_matrix
```

이 단계는 FA 분석이 검증된 뒤에 진행한다. 아직 검증되지 않은 점수체계를 바로 백테스트에 넣으면, 성과가 좋아도 왜 좋은지 알 수 없고 성과가 나빠도 어디를 고쳐야 할지 알 수 없다.

### 분기별 반복 검증 루프

위 4단계는 한 번만 실행하는 절차가 아니다. 분기 FA는 새로운 정기보고서가 나올 때마다 같은 순서로 다시 실행되어야 한다.

따라서 자동화의 기본 단위는 다음과 같다.

```text
TARGET_QUARTER = YYYYQn
```

분기별 실행 흐름은 다음처럼 정의한다.

```text
1. 원천 데이터 수집
2. company_quarter_fa 생성
3. 데이터 품질 검증
4. 점수 계산 검증
5. 전분기 대비 변화 검증
6. 시점 안전성 검증
7. 검증 산출물 저장
```

각 실행은 단순히 점수 CSV를 만드는 데서 끝나면 안 된다. 점수와 함께 검증 결과도 남겨야 한다.


| 산출물                                | 역할                      |
| ---------------------------------- | ----------------------- |
| `company_quarter_fa_YYYYQn.csv`    | 해당 분기의 기업별 FA 원장        |
| `fa_validation_YYYYQn.md`          | 데이터 품질, 점수 분포, 경고 사항 요약 |
| `fa_score_distribution_YYYYQn.csv` | 섹터별 점수 분포와 분위수          |
| `fa_score_change_YYYYQn.csv`       | 전분기 대비 점수 변화와 상위/하위 이동  |
| `fa_excluded_YYYYQn.csv`           | 제외 종목과 제외 사유            |


분기 검증에서 최소로 확인할 항목은 다음과 같다.


| 검증 영역    | 확인 질문                                                     |
| -------- | --------------------------------------------------------- |
| 데이터 커버리지 | 이번 분기 종목 수와 핵심 재무지표 결측률이 기대 범위인가                          |
| 날짜 정합성   | 모든 행에서 `period_end < available_date`가 성립하는가               |
| 시점 안전성   | 검증 대상 리밸런싱 날짜에서 `available_date <= rebalance_date`가 성립하는가 |
| 점수 분포    | 섹터별 점수가 한쪽으로 과도하게 몰리지 않았는가                                |
| 산식 방향성   | 좋은 원지표가 높은 서브스코어로 연결되는가                                   |
| 신뢰도      | `score_confidence`가 낮은 종목이 상위권에 과도하게 포함되지 않았는가            |
| 변화 설명    | 전분기 대비 점수 급등락이 재무 변화나 위험 변화로 설명되는가                        |
| 제외 사유    | 제외 종목이 조용히 버려지지 않고 `excluded_reason`에 남았는가                |


검증 결과는 `pass`, `warning`, `fail` 중 하나로 남긴다.


| 판정        | 의미                       | 처리                 |
| --------- | ------------------------ | ------------------ |
| `pass`    | 자동화 결과를 다음 단계에서 사용할 수 있음 | 리밸런싱 스냅샷 생성 가능     |
| `warning` | 결과는 생성됐지만 공식 갱신에는 신중해야 함 | 결과를 저장하되 기본적으로 마지막 정상 결과 유지 |
| `fail`    | 점수 또는 데이터가 신뢰하기 어려움      | 백테스트와 종목선정 연결 중단   |


v1의 자동화는 `fail` 기준을 너무 복잡하게 만들지 않는다. 우선 아래 조건을 실패 후보로 둔다.


| 실패 후보                                      | 이유                  |
| ------------------------------------------ | ------------------- |
| 핵심 재무지표 결측률이 직전 분기보다 급격히 증가                | 데이터 수집 또는 매핑 문제 가능성 |
| 특정 섹터의 점수 존재 종목 수가 지나치게 적음                 | 섹터 상대평가가 불안정함       |
| `available_date`가 `period_end`보다 빠른 행이 존재  | 시점 오류 가능성           |
| 상위 점수군에 `score_confidence`가 낮은 종목이 과도하게 포함 | 결측 재정규화로 인한 왜곡 가능성  |
| 전분기 대비 점수 급변이 원지표 변화로 설명되지 않음              | 산식 또는 데이터 오류 가능성    |


중요한 것은 매 분기 “점수가 나왔다”가 아니라, **이번 분기 점수를 다음 단계에 써도 되는지 판단하는 검증 기록이 남는 것**이다.

### FA 분석 무개입 자동화 목표

이 프로젝트의 FA 자동화는 사람이 매 분기 노트북을 열어 수동 판단하는 방식을 목표로 하지 않는다.

목표는 다음과 같다.

```text
새 정기보고서가 들어오면
시스템이 수집, 계산, 검증, 판정, 저장까지 자동으로 수행하고,
사람이 개입하지 않아도 정해진 기본 행동으로 마무리한다.
```

더 정확히 말하면 v1의 목표는 **매매 무개입**이 아니라 **FA 분석 무개입**이다.

다만 무개입 자동화의 방향은 공격적인 자동 갱신이 아니다. 기본 철학은 **하방 방어**다.

```text
확실한 결과만 다음 단계로 넘긴다.
애매한 결과는 보류한다.
실패한 결과는 격리한다.
```

즉 자동화는 새로운 점수를 계속 밀어붙이는 장치가 아니라, 신뢰할 수 없는 점수가 리밸런싱과 백테스트에 흘러 들어가지 않도록 막는 방어선이다.

이렇게 정의하는 이유는 FA 점수의 역할 때문이다.

FA 점수는 단기 매수 신호가 아니라, 리밸런싱과 백테스트가 참조하는 기초 입력값이다. 이 입력값이 잘못되면 뒤 단계의 종목선정, 비중 배분, 성과 검증이 모두 왜곡된다. 따라서 자동화에서 가장 위험한 상태는 “새 점수를 만들지 못한 상태”가 아니라, **신뢰하기 어려운 새 점수를 정상 결과처럼 downstream에 넘기는 상태**다.

특히 분기 자동화에서는 아래 문제가 반복적으로 발생할 수 있다.


| 위험              | 왜 하방 방어가 필요한가                          |
| --------------- | -------------------------------------- |
| 공시 지연           | 아직 투자자가 알 수 없는 정보를 사용하면 룩어헤드가 생긴다      |
| 재무제표 결측         | 일부 지표만으로 재정규화된 점수가 과대평가될 수 있다          |
| WICS/DART 매핑 오류 | 다른 기업의 재무값이 붙거나 분석 대상이 누락될 수 있다        |
| 섹터 표본 부족        | 상대평가 percentile이 불안정해질 수 있다            |
| 점수 급변           | 실제 실적 변화가 아니라 데이터 오류나 계정 추출 오류일 수 있다   |
| 자동 리밸런싱 연결      | 한 번 잘못된 FA 입력이 다음 단계 전체 포트폴리오를 바꿀 수 있다 |


그래서 기본 정책은 다음 순서로 둔다.

```text
수익 기회를 놓치는 위험 < 잘못된 입력으로 하방이 무너지는 위험
```

이 프로젝트에서 FA 자동화의 1차 책임은 수익률을 공격적으로 높이는 것이 아니라, **검증되지 않은 점수가 투자 판단에 사용되는 것을 막는 것**이다. `hold_previous`와 `quarantine`을 기본값으로 두는 이유도 여기에 있다.


| 구분            | v1 목표 여부 | 설명                                |
| ------------- | -------- | --------------------------------- |
| FA 데이터 수집 자동화 | 포함       | 정기보고서, WICS, 시장 스냅샷을 자동 갱신한다      |
| FA 점수 계산 자동화  | 포함       | `company_quarter_fa`와 점수를 자동 생성한다 |
| FA 검증 자동화     | 포함       | 데이터 품질과 점수 신뢰도를 자동 판정한다           |
| 리밸런싱 입력 생성    | 포함       | `pass`인 경우 다음 단계 입력을 자동 생성한다      |
| 주문 실행 자동화     | 제외       | 실제 매매는 별도 승인, 리스크 관리, 운영 체계가 필요하다 |


따라서 v1부터 아래 원칙을 따른다.


| 원칙       | 설명                                                              |
| -------- | --------------------------------------------------------------- |
| 설정 파일 기반 | 분석 연도, 분기, 섹터, 임계값, 제외 규칙은 코드 수정이 아니라 설정값으로 관리한다                |
| 재실행 가능성  | 같은 입력으로 같은 분기를 다시 실행하면 같은 산출물이 나와야 한다                           |
| 증분 실행    | 새 분기가 들어오면 전체를 매번 손으로 다시 만들지 않고 해당 분기와 필요한 비교 구간만 갱신한다          |
| 자동 판정    | 결측률, 날짜 정합성, 점수 분포, 신뢰도 기준으로 `pass`, `warning`, `fail`을 자동 부여한다 |
| 하방 방어    | `pass`가 아니면 새 결과를 적극적으로 사용하지 않는 것을 기본값으로 둔다                     |
| 자동 처리    | 판정 결과별 기본 행동을 미리 정해 사람이 없어도 보수적으로 파이프라인이 끝난다                    |
| 자동 중단    | `fail`이면 리밸런싱, 종목선정, 백테스트 연결을 자동으로 막고 이전 유효 결과를 보존한다            |
| 무개입 운영   | 사람은 운영 중 필수 판단자가 아니라, 나중에 필요하면 로그를 보는 관찰자다                      |
| 로그 보존    | 실행 시각, 입력 파일, 생성 행 수, 제외 종목 수, 검증 판정을 기록한다                      |


운영 관점의 전체 흐름은 다음과 같다.

```text
정기 실행 스케줄
→ 원천 데이터 갱신 확인
→ 이번 분기 자동 판별
→ company_quarter_fa 생성
→ score 계산
→ validation rules 실행
→ pass/warning/fail 판정
→ 산출물과 로그 저장
→ 판정별 기본 행동 실행
```

자동화 설정은 별도 파일로 분리하는 것이 좋다.

예시:

```text
fa_automation_config
- target_universe
- target_sectors
- quarter_schedule
- metric_required_columns
- missing_rate_threshold
- score_confidence_threshold
- top_bucket_confidence_threshold
- warning_fallback_policy
- output_paths
```

설정 변수는 이름만 두면 나중에 추적하기 어렵다. 따라서 각 변수는 “어디서 온 값인지”와 “어느 검증에서 쓰이는지”를 함께 문서화한다.


| 설정 변수                             | 값의 출처                  | 참조 데이터/컬럼                                                                           | 쓰이는 곳                                 |
| --------------------------------- | ---------------------- | ----------------------------------------------------------------------------------- | ------------------------------------- |
| `target_universe`                 | 투자/분석 대상 정의            | WICS 종목 목록, 상장시장, 제외 종목 규칙                                                          | 분석 대상 기업 필터                           |
| `target_sectors`                  | WICS 분석 범위             | `wics_large`, `wics_large_code`                                                     | 섹터별 상대평가와 리포트                         |
| `quarter_schedule`                | DART 정기보고서 주기와 리밸런싱 정책 | `report_code`, `period_end`, `available_date`, `rebalance_date`                     | 새 분기 감지와 시점 검증                        |
| `metric_required_columns`         | 점수 산식에 필요한 최소 지표       | `revenue`, `operating_income`, `net_income`, `total_equity`, `operating_cashflow` 등 | 데이터 커버리지와 제외 사유 판정                    |
| `missing_rate_threshold`          | 과거 검증 리포트와 허용 결측률 정책   | 섹터별/분기별 핵심 컬럼 결측률                                                                   | `pass`, `warning`, `fail` 판정          |
| `score_confidence_threshold`      | 점수 계산에 실제 사용된 지표 비율    | `level_score`, `change_score`, `risk_penalty` 하위 지표 결측 여부                           | 낮은 신뢰도 점수 차단                          |
| `top_bucket_confidence_threshold` | 상위 점수군의 최소 신뢰도 정책      | 상위 20%, 상위 N개 종목의 `score_confidence`                                                | 상위 종목 왜곡 방지                           |
| `score_distribution_threshold`    | 과거 정상 분기의 점수 분포        | 섹터별 `fa_score` 평균, 표준편차, 분위수                                                        | 점수 쏠림과 변별력 검증                         |
| `score_change_threshold`          | 전분기 대비 변화 허용 범위        | `fa_score`, `level_score`, `change_score`, `risk_penalty` 전분기 차이                    | 비정상 급변 탐지                             |
| `fallback_policy`                 | 하방 방어 운영 원칙            | `pass`, `warning`, `fail` 판정 결과                                                     | `hold_previous`, `quarantine` 등 기본 행동 |
| `warning_fallback_policy`         | 경고 발생 시 하방 방어 정책        | `warning` 판정 결과와 rule severity                                                     | 마지막 정상 결과 유지, 격리, 건너뛰기 등            |
| `output_paths`                    | 산출물 저장 규칙              | 분기별 CSV, Markdown, 로그 경로                                                            | 재실행성과 추적성 확보                          |


예를 들어 `score_confidence_threshold`는 임의 숫자가 아니라, “이번 점수가 실제 사용 가능한 하위 지표를 얼마나 많이 기반으로 계산되었는가”를 제한하기 위한 값이다. 따라서 이 값은 점수 산식의 하위 지표 목록과 결측 여부에서 나온다.

마찬가지로 `missing_rate_threshold`는 단일 컬럼 하나가 아니라, 핵심 재무 컬럼들의 분기별 결측률에서 나온다. 이 기준을 넘으면 데이터가 부족한 상태로 판단하고 새 결과를 적극적으로 사용하지 않는다.

노트북의 역할도 바뀐다.


| 구분           | 역할                           |
| ------------ | ---------------------------- |
| 자동화 스크립트     | 매 분기 실제 수집, 계산, 검증, 판정을 실행   |
| 검증 노트북       | 자동화 결과를 사람이 필요할 때 점검하는 분석 화면 |
| Markdown 리포트 | 자동화가 남기는 공식 검증 결과            |


즉 노트북은 운영의 중심이 아니라, 자동화 결과를 들여다보는 보조 도구다. 운영의 중심은 재실행 가능한 스크립트와 검증 규칙이어야 한다.

최종 운영 형태는 다음처럼 단순해야 한다. 아래 명령은 목표 형태를 설명하는 예시이며, 실제 파일명은 구현 단계에서 확정한다.

```powershell
python etl/wics_dart/pipeline/run_fa_quarterly_pipeline.py --auto
```

이 명령은 아래 작업을 한 번에 수행해야 한다.

```text
1. 새로 처리할 분기가 있는지 확인
2. 필요한 원천 데이터 갱신
3. FA 원장 생성
4. 점수 계산
5. 검증 규칙 실행
6. pass/warning/fail 판정
7. 산출물과 로그 저장
8. 판정별 기본 행동 실행
9. 실행 로그와 요약 저장
```

스케줄러에 붙일 때는 사람이 실행하지 않아도 된다.

```text
매일 또는 매주 정해진 시간 실행
→ 새 데이터가 없으면 종료
→ 새 데이터가 있으면 자동 처리
→ 문제가 있어도 기본 정책에 따라 자동 종료
```

무개입 운영에서는 판정별 기본 행동을 먼저 정한다.


| 판정        | 기본 행동                                |
| --------- | ------------------------------------ |
| `pass`    | 이번 분기 결과를 공식 결과로 확정하고 다음 단계 입력을 생성한다 |
| `warning` | 이번 분기 결과는 저장하되 기본적으로 공식 갱신하지 않는다     |
| `fail`    | 이번 분기 결과를 격리하고 다음 단계에는 전달하지 않는다      |


`warning`과 `fail`도 사람이 즉시 처리해야 하는 상태가 아니다. 자동화는 하방 방어를 위해 아래 fallback 정책 중 하나를 설정으로 선택해야 한다.


| fallback 정책       | 설명                                    |
| ----------------- | ------------------------------------- |
| `hold_previous`   | 마지막 `pass` 결과를 유지한다. v1 기본값으로 둔다      |
| `skip_quarter`    | 이번 분기 리밸런싱 입력 생성을 건너뛴다                |
| `quarantine_only` | 결과를 저장만 하고 어떤 downstream 단계에도 넘기지 않는다 |
| `notify_only`     | 요약을 남기지만 파이프라인은 정해진 정책대로 종료한다         |


v1 기본 정책은 다음처럼 둔다.


| 판정        | 기본 fallback                         |
| --------- | ----------------------------------- |
| `pass`    | 신규 결과 사용                            |
| `warning` | `hold_previous`                     |
| `fail`    | `quarantine_only` + `hold_previous` |


이렇게 하면 자동화가 멈추지 않으면서도, 불완전한 점수가 새로운 판단으로 쓰이는 것을 막을 수 있다.

이 원칙은 투자 성과를 공격적으로 개선하기 위한 것이 아니라, 잘못된 데이터나 불안정한 점수로 인해 하방이 무너지는 것을 막기 위한 운영 원칙이다.

사람이 손대는 일은 운영 중 필수 과정이 아니라, 기준을 바꾸고 싶을 때 발생하는 별도 개선 작업이다.


| 상황                           | 사람의 역할                            |
| ---------------------------- | --------------------------------- |
| 기준을 바꾸고 싶음                   | 설정과 검증 기준을 업데이트                   |
| 새 섹터 또는 새 지표 추가              | 산식과 검증 규칙을 설계                     |
| 반복적으로 `warning` 또는 `fail` 발생 | 로그를 보고 자동화 기준을 개선                 |
| 예외 기업 처리 정책 변경               | 예외 규칙과 `excluded_reason` 체계를 업데이트 |


따라서 최종 목표는 “매 분기 사람이 분석을 다시 하는 것”도 아니고 “예외가 날 때마다 사람이 처리하는 것”도 아니다. 목표는 **매 분기 자동화가 분석과 예외 처리를 끝내고, 사람은 필요할 때만 사후 개선을 하는 구조**다.

## 13. v1에서 하지 않을 것

처음부터 너무 많은 요소를 넣으면 검증이 어려워진다.

v1에서는 다음을 핵심 범위에서 제외한다.


| 제외 항목        | 이유                   |
| ------------ | -------------------- |
| 뉴스 감성 분석     | 시점 관리와 해석 기준이 복잡함    |
| LLM 주관 평가 점수 | 백테스트 재현성이 낮아질 수 있음   |
| TA 진입/청산 조건  | FA 점수의 유효성을 먼저 봐야 함  |
| 복잡한 비중 최적화   | 점수 검증 전에는 과최적화 위험이 큼 |
| 실시간 자동매매     | FA 원장과 백테스트 검증 이후 단계 |


LLM은 v1에서 점수를 직접 만들기보다, 점수 결과를 설명하거나 공시 내용을 요약하는 보조 계층으로 두는 편이 안전하다.

## 14. 최종 정의

FA 자동화의 v1 목표는 다음이 아니다.

```text
자동으로 좋은 종목을 매수한다
```

v1 목표는 다음이다.

```text
분기마다 기업의 재무 상태, 변화, 위험을 같은 기준으로 계산하고,
그 결과를 백테스트가 사용할 수 있는 시점 안전한 데이터로 만든다.
```

따라서 지금 우선순위는 리밸런싱 규칙이 아니라 다음 세 가지다.

```text
1. company_quarter_fa 테이블 정의
2. quarter_fa_score 점수 구조 정의
3. available_date 기준의 시점 검증
```

이 세 가지가 검증되면 그 다음에 리밸런싱, 종목 선택, vectorBT 백테스트로 연결한다.
