# FA 학습가이드

## 1. 먼저 잡아야 할 큰 그림

이 프로젝트에서 FA는 종목을 바로 사라는 신호가 아니다.

FA의 역할은 다음 한 문장이다.

```text
분기마다 기업의 재무 상태가 좋아졌는지, 나빠졌는지, 위험해졌는지 점수화한다.
```

그 다음에야 백테스트가 이 점수를 사용한다.

```text
DART/WICS 데이터
→ 분기별 FA 점수
→ 검증
→ 리밸런싱
→ vectorBT 백테스트
```

---

## 2. 꼭 알아야 할 세 날짜

FA 자동화에서 가장 중요한 것은 날짜를 섞지 않는 것이다.


| 개념               | 의미                    |
| ---------------- | --------------------- |
| `fiscal_quarter` | 재무제표가 설명하는 분기         |
| `available_date` | 그 재무제표를 실제로 알 수 있는 날짜 |
| `rebalance_date` | 포트폴리오를 다시 짜는 날짜       |


백테스트에서는 항상 이 조건을 지켜야 한다.

```text
available_date <= rebalance_date
```

이걸 어기면 미래 정보를 보고 투자한 것이 되어 결과를 믿을 수 없다.

## 3. 점수 구조

v1 점수는 세 덩어리로 본다.

```text
fa_score = level_score + change_score - risk_penalty
```


| 점수             | 의미               |
| -------------- | ---------------- |
| `level_score`  | 지금 재무 상태가 좋은가    |
| `change_score` | 전년 동기보다 좋아지고 있는가 |
| `risk_penalty` | 피해야 할 위험이 있는가    |


좋은 점수체계는 현재 좋은 기업과 좋아지는 기업을 구분해야 한다.

점수는 단순히 지표를 더하는 방식이 아니다.

먼저 재무제표 원천값에서 파생 지표를 만든다.


| 파생 지표                         | 의미                |
| ----------------------------- | ----------------- |
| `operating_margin`            | 영업이익률             |
| `roe`                         | 자기자본이익률           |
| `debt_ratio`                  | 부채비율              |
| `current_ratio`               | 유동비율              |
| `ocf_to_revenue`              | 매출 대비 영업현금흐름      |
| `per_proxy`, `pbr_proxy`      | 시가총액 기반 밸류에이션 프록시 |
| `revenue_growth_yoy`          | 전년 동기 대비 매출 성장률   |
| `operating_income_growth_yoy` | 전년 동기 대비 영업이익 성장률 |


그 다음 같은 분기, 같은 WICS 섹터 안에서 percentile로 바꾼다.

```text
높을수록 좋은 지표 → percentile_rank(metric)
낮을수록 좋은 지표 → percentile_rank(-metric)
```

마지막으로 하위 점수를 배점에 맞게 합산한다.

```text
level_score  = 수익성 + 안정성 + 현금흐름 + 밸류에이션
change_score = 매출 성장 + 영업이익 성장 + 수익성 개선 + 현금흐름 개선
fa_score     = level_score + change_score - risk_penalty
```

결측은 0점과 다르다. 0점은 나쁜 값이고, 결측은 판단 불가다. 따라서 결측이 있으면 계산 가능한 항목만 재정규화하되 `score_confidence`를 낮춘다.

## 4. 좋은 점수인지 판단하는 법

점수가 좋고 나쁨은 네 단계로 판단한다.


| 단계     | 질문                            |
| ------ | ----------------------------- |
| 데이터 검증 | 데이터가 충분하고 결측이 적은가             |
| 산식 검증  | 점수가 의도한 방향으로 계산되는가            |
| 상식 검증  | 상위/하위 종목이 사람이 봐도 납득되는가        |
| 성과 검증  | 상위 점수군이 이후 수익률에서 하위 점수군보다 나은가 |


처음부터 백테스트 성과만 보면 안 된다.

성과가 안 좋아도 데이터가 틀린 것인지, 점수 산식이 틀린 것인지, 리밸런싱 연결이 틀린 것인지 알 수 없기 때문이다.

## 6. 지금 단계의 결론

현재 우선순위는 백테스트가 아니다.

먼저 해야 할 일은 이것이다.

```text
1. 한 섹터를 고른다.
2. 점수 분포를 본다.
3. 상위/하위 종목을 확인한다.
4. 결측과 점수 신뢰도를 본다.
5. 이후 수익률을 붙여본다.
```

이 과정을 통과하면 그때 vectorBT 백테스트로 연결한다.

## 7. 분기마다 반복할 일

FA 자동화는 한 번 점수를 만들고 끝나는 작업이 아니다.

매 분기 새 재무제표가 들어오면 같은 검증을 반복해야 한다. 다만 목표는 사람이 매번 노트북을 열어 손으로 판단하는 것이 아니다.

목표는 이것이다.

```text
시스템이 매 분기 자동으로 계산하고,
자동으로 검증하고,
자동으로 pass, warning, fail을 남긴다.
```

이 단계의 목표는 **FA 분석 무개입 자동화**다.

```text
사람이 매번 분석하지 않는다.
시스템이 분석하고, 예외도 정해진 규칙대로 처리한다.
```

다만 이것은 아직 주문 실행까지 자동화한다는 뜻은 아니다. 지금 목표는 매매 자동화가 아니라, 리밸런싱과 백테스트가 믿고 쓸 수 있는 FA 입력값을 자동으로 만드는 것이다.

이때 자동화의 기본 방향은 공격이 아니라 **하방 방어**다.

```text
확실하면 갱신한다.
애매하면 이전 정상 결과를 유지한다.
실패하면 새 결과를 쓰지 않는다.
```

즉 시스템은 좋은 종목을 억지로 계속 새로 고르는 장치가 아니라, 믿기 어려운 점수가 다음 단계로 넘어가지 않도록 막는 방어선이다.

왜 이렇게 보수적으로 잡는가?

FA 점수는 백테스트와 리밸런싱의 입력값이다. 이 입력값이 틀리면 뒤의 결과가 전부 흔들린다. 그래서 새 점수를 못 쓰는 것보다 더 위험한 것은, 틀렸을 수 있는 새 점수를 정상 결과처럼 쓰는 것이다.

예를 들어 이런 일이 생길 수 있다.


| 상황                   | 위험                      |
| -------------------- | ----------------------- |
| 공시가 아직 사용 가능한 시점이 아님 | 미래 정보를 보고 투자한 것처럼 된다    |
| 주요 재무지표가 비어 있음       | 일부 지표만으로 높은 점수가 나올 수 있다 |
| 매핑이 틀림               | 다른 기업의 재무값을 사용할 수 있다    |
| 점수가 갑자기 바뀜           | 실적 변화가 아니라 데이터 오류일 수 있다 |


그래서 기본 원칙은 이것이다.

```text
기회를 조금 놓치더라도,
믿기 어려운 점수로 하방이 무너지는 것을 먼저 막는다.
```

```text
1. 이번 분기 FA 원장을 만든다.
2. 데이터 결측과 날짜 정합성을 확인한다.
3. 점수 분포와 상위/하위 종목을 확인한다.
4. 전분기 대비 점수 변화가 설명되는지 본다.
5. pass, warning, fail 중 하나로 검증 결과를 남긴다.
```

중요한 질문은 이것이다.

```text
이번 분기 점수를 리밸런싱과 백테스트에 써도 되는가?
```

따라서 노트북은 예쁜 리포트가 아니라, 매 분기 같은 질문에 답하는 검증 도구로 발전시킨다.

장기적으로는 노트북도 운영의 중심이 아니다.

```text
자동화 스크립트가 계산과 검증을 실행하고,
노트북은 사람이 나중에 원인을 살펴보고 싶을 때 여는 화면이 된다.
```

`warning`이나 `fail`이 나도 사람이 즉시 손대는 구조로 만들지 않는다.


| 판정        | 자동 처리                           |
| --------- | ------------------------------- |
| `pass`    | 이번 분기 결과를 다음 단계 입력으로 넘긴다        |
| `warning` | 결과는 저장하지만 기본적으로 마지막 정상 결과를 유지한다 |
| `fail`    | 결과를 격리하고 새 입력으로 쓰지 않는다          |


기본값은 아래처럼 둔다.


| 판정        | 기본 행동                |
| --------- | -------------------- |
| `pass`    | 신규 결과 사용             |
| `warning` | 마지막 정상 결과 유지         |
| `fail`    | 결과 격리 + 마지막 정상 결과 유지 |


사람이 손대는 경우는 운영 중 필수 처리라기보다, 기준을 바꾸고 싶을 때의 개선 작업이다.


| 상황             | 사람의 역할              |
| -------------- | ------------------- |
| 같은 경고가 반복됨     | 로그를 보고 자동화 기준을 개선한다 |
| 새 지표 추가        | 설정과 검증 기준을 업데이트한다   |
| 예외 기업 처리 방식 변경 | 제외 규칙을 업데이트한다       |


즉 최종 목표는 분석을 반복하는 것이 아니라, **분석 반복 자체를 자동화하는 것**이다.

최종 사용 방식은 아래처럼 단순해야 한다.

```text
정해진 시간에 자동 실행
→ 새 데이터가 없으면 종료
→ 새 데이터가 있으면 계산과 검증
→ pass면 다음 단계 입력 생성
→ warning 또는 fail이면 하방 방어 정책으로 자동 종료
```

---

최종 목표는 `fa_score`로 교체하는 것이다.
